## 🤖 Les agents

6 agents spécialisés. Chacun a une responsabilité unique et n'empiète pas sur les autres.

| Agent | Rôle | Modèle |
|---|---|---|
| **Oriane** | PM Orchestrator — orchestre les workflows produit de bout en bout | Opus |
| **Ariane** | Architecte — décisions techniques justifiées, analyse d'impact, planning | Opus |
| **Diane** | UX Designer — design system, flows, accessibilité, copywriting, audits | Opus |
| **Eva** | Impact Evaluator — évaluation multi-dimensionnelle des décisions | Opus |
| **Justine** | Challenger — revue adversariale de chaque livrable | Opus |
| **Claire** | Clarity Challenger — questions Socratiques jusqu'à zéro ambiguïté | Default |

## 🔁 Boucle d'exécution — pour chaque skill

```
[Skill exécutée]
      ↓
Justine challenge le livrable (complétude, cohérence, angles morts)
      ↓
Eva évalue l'impact si conséquences larges (skip si purement descriptif)
      ↓
Blockers résolus avant présentation
      ↓
Livrable présenté → "Do you approve?" — STOP. Pas de suite sans approbation.
      ↓ si non approuvé → fix → Justine re-challenge
      ↓ si approuvé
Réconciliation amont (déduplication + propagation des contraintes)
      ↓
[Skill suivante]
```

## 🆕 Nouveautés depuis v3.7.0-pm.1

- **Skill `spike`** — investigation time-boxée avant de s'engager sur une décision. Disponible chez Oriane, Ariane et Diane.
- **Challenge gates mécaniques** — chaque skill vérifie automatiquement structure, complétude et format avant présentation.
- **Réconciliation amont bidirectionnelle** — déduplication du contenu migré + propagation des nouvelles contraintes vers les docs amont.
- **4 skills UX brownfield** — `ux-design-system-update`, `ux-flow-update`, `ux-accessibility-update`, `ux-copywriting-update`.
- **Simplification des agents** — responsabilité unique stricte, alignement avec les cours.
- **Langage business dans `pm-constitution`** — pas de jargon technique dans la vision ou les contraintes.

## 💬 C'est une beta — challenge bienvenu

Cette release expérimente une approche différente de ce qu'on fait habituellement dans le framework : **on a misé sur des agents et des skills plutôt que des commandes**.

L'idée est que chaque agent (Oriane, Ariane, Diane...) orchestre ses propres skills de façon autonome, avec des gates de challenge intégrées, plutôt que de tout piloter via des commandes SDLC classiques.

**Rien n'est ancré.** Si tu trouves que le découpage ne tient pas, que les agents débordent, que les gates ralentissent sans apporter de valeur, que l'approche skills vs commandes ne fonctionne pas en pratique — dis-le. On itère.

Ce qui nous intéresse particulièrement :
- Est-ce que le flow greenfield/brownfield tient sur un vrai projet ?
- Est-ce que les challenge gates (Justine) apportent de la valeur ou bloquent inutilement ?
- Est-ce que la séparation des responsabilités entre agents est claire à l'usage ?
- Est-ce que l'approche "skills orchestrés par agents" est plus ou moins lisible que des commandes directes ?

Signaler ici : https://github.com/ai-driven-dev/aidd-framework/issues
